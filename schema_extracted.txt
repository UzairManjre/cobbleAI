1. MongoDB Collections
MongoDB stores all application state. Every document uses UUID strings as _id (not ObjectId) for portability. All timestamps are stored in UTC. Motor (async driver) is used exclusively — no synchronous PyMongo calls in request handlers.

Eviction strategy — detail
Hard cap: 40 turns (20 exchanges). On turn 41, oldest 10 turns are pruned.
Token cap: if token_count exceeds 6 000, oldest turns pruned until under limit regardless of turn count.
Before pruning: the LLM generates a 2–3 sentence summary of the turns being evicted. This summary is stored in the summary field and prepended to the next prompt.
Pruned turns are appended to cold_chat_archive (§1.7) before deletion from this collection.

3. Redis Key Schema
Redis serves four separate roles. Each role uses a dedicated logical database (DB number) to prevent key collisions and allow independent flush operations.
3.1 DB 0 — Celery Broker (managed by Celery)
Celery manages all keys in DB 0 automatically. Do not read or write DB 0 manually. Keys follow Celery's internal format: celery, celery-task-meta-{task_id}, _kombu.binding.celery, etc.
3.2 DB 1 — Celery Result Backend (managed by Celery)
Celery stores task results in DB 1. Query via Celery's AsyncResult API, never directly. Results auto-expire after 24 hours (CELERY_RESULT_EXPIRES = 86400).

4.4 Retrieval Strategy
Two-stage retrieve-then-rerank on every /chat request:
Stage 1 — ANN search: query the course collection with the embedded user message. Retrieve top 10 candidates by cosine similarity. Apply has_image = false filter unless the course has no text chunks.
Stage 2 — Rerank: pass (query_text, chunk_text) pairs to the cross-encoder/ms-marco-MiniLM-L-6-v2. Score each pair jointly. Keep top 3 by cross-encoder score.
Inject top 3 chunks into the LLM prompt. Each chunk is labeled with its original_filename and page_number for inline citations.
Store all 10 candidate chunk_ids and the 3 final chunk_ids in the ChatTurn.rag_chunk_ids field for auditability.

SECURITY   
All S3 objects are private. The API never serves S3 URLs directly. Files are served via pre-signed URLs with a 15-minute expiry, generated server-side only for authenticated users with access to the relevant course.
6. Schema Versioning & Migration Strategy
6.1 MongoDB
MongoDB is schemaless but all writes go through Pydantic models — schema is enforced at the application layer, not the DB layer.
Use a db_version collection with one document { version: N, applied_at: datetime } to track schema migrations.
Migration scripts live in /migrations/mongo/NNNN_description.py. Each script is idempotent — safe to run twice.
Run migrations at API startup before accepting traffic. If migration fails, the API refuses to start.
6.2 Qdrant
Qdrant collections are created by the ingestion worker on first document upload for a course.
If the embedding model changes (e.g. upgrade from MiniLM-L6-v2 to a multimodal model), all collections must be re-indexed. This is a bulk Celery task: re_embed_course.delay(course_id) for each course.
Collection names include no version suffix — re-indexing drops and recreates the collection. Documents are marked processing_status = "pending" and re-queued.
6.3 Redis
Redis holds no persistent application state — all persistent data is in MongoDB or S3.
Cache keys include a version prefix (e.g. v1:concept_map:{course_id}) so a cache invalidation strategy change does not serve stale data during rolling deployments.
On deployment, flush DB 2 (cache) and DB 3 (rate limits) — this is safe because neither holds durable state.
